- # [Introducing the Stateful Runtime Environment for Agents in Amazon Bedrock](https://openai.com/index/introducing-the-stateful-runtime-environment-for-agents-in-amazon-bedrock/)
	- OpenAI **company announcement** (Feb 27, 2026): joint work with **Amazon** on a **Stateful Runtime Environment** that runs **inside Amazon Bedrock**, powered by **OpenAI models**, tuned for **AWS** and **agentic workflows**—positioned as reducing the custom glue teams build around **stateless APIs** (state storage, tool invocation, error handling, resuming long tasks) by carrying **working context** across steps: memory/history, tool and workflow state, environment use, and **identity/permission** boundaries.
	- Stated goals: faster path to **production** multi-step work (support, sales ops, IT automation, finance with approvals/audits), **long-horizon** reliability, and **AWS-native** deployment/governance aligned with existing security posture and tooling. At announcement time, availability was **“will be available soon”** with contact/OpenAI sales funnel for interest.
	- **How this sits next to [[e2b]] (if applicable)**
		- **Complementary layers, not the same product**: this runtime is framed as **Bedrock-native orchestration + durable state + governance** for **multi-step agents** using OpenAI models on AWS. [[e2b]] is a **separate product category** focused on **ephemeral isolated execution** (code, browser, desktop sandboxes, notebooks, etc.)—closer to **sandbox compute** than to **managed conversation/workflow state inside Bedrock**.
		- **Possible architecture**: Bedrock + Stateful Runtime for **planning, policy, and step continuity**; **E2B** (or self-hosted [[AWS/GitHub/sample-e2b-on-aws]]) when you need **strong isolation for arbitrary code or desktop/browser tool use** the announcement does not describe as the core primitive.
		- **Vs [[AWS/Lightsail]] / [[AWS/ECS/Express Mode]]**: those address **general hosting** (VPS) and **simplified long-running container services**; the announcement is about a **Bedrock agent runtime**, not replacing how you host a web tier—though any full solution might still use ECS, Lambda, etc. alongside Bedrock.
	- **How this sits next to [[GitHub/Codespace]] (if applicable)**
		- **Different product class**: Codespaces is a **GitHub‑centric cloud dev environment** ([[DevContainer]] on **Azure**): IDE/repo workflow, human (or agent **inside** that workspace) software development, billing per codespace, strong **git + PR** integration.
		- The Bedrock **Stateful Runtime** is framed as **production agent orchestration and durable state inside AWS/Bedrock**, not as a replacement for **VS Code in the browser** or **repo‑bound devcontainers**.
		- **Where they rhyme**: both care about **persistence across steps** and **long‑running** work (Codespaces keeps running after disconnect; the runtime keeps **agent working context**). **Where they don’t**: tenancy model (per‑developer workspace vs Bedrock‑managed agent runtime), cloud (GitHub/Azure vs customer **AWS** governance story), and primary user (engineering **dev loop** vs **enterprise agent workflows** on Bedrock).
