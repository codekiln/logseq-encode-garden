# Git-friendly task manager comparison
	- ## Evaluation criteria
		- A good replacement for this repo should store task state in Git-friendly text files or Git-native objects, avoid [[SQLite]] as the committed source of truth, and work from a CLI without requiring a hosted service.
		- It should be legible to AI coding agents without dumping huge JSONL blobs into context, support basic issue states and dependencies, and have a clear ready/blocked view for resuming work.
	- ## Candidates
		- {{embed [[tk]]}}
	- ## [[BeansTasks]]
		- Source: [hmans/beans](https://github.com/hmans/beans).
		- [[BeansTasks]] is a CLI-based, flat-file issue tracker that stores project tasks as Markdown files in `.beans/` and aims at both humans and coding agents.
		- What it supports:
			- CLI task creation, listing, viewing, updating, and archiving.
			- A terminal UI for browsing and managing work.
			- Agent-oriented context through `beans prime`.
			- A GraphQL query engine for selective task context.
			- Archived completed beans as project memory.
			- Roadmap generation from task data.
		- Evaluation:
			- It is attractive if the desired workflow is closer to "agent-aware product manager for a repo" than "minimal ticket queue".
			- Plain Markdown storage is a strong fit for Git and for this garden's text-first sensibility.
			- Its own docs warn that it is under heavy development and that schema changes may require manual migrations, so stability risk is higher than the feature list suggests.
		- What migration would take here:
			- Preserve `.beads/issues.jsonl` and decide whether the two closed Beads issues should become archived beans or be left as Beads history.
			- Run `beans init` to create `.beans/` and `.beans.yml`.
			- Manually create or agent-convert the open `bd-10` issue into a bean, because the migration path is less direct than `tk migrate-beads`.
			- Run `beans help`, `beans prime`, and the listing command to confirm agents can see the open work.
			- Update Rulesync-managed instructions to use `beans prime` and Beans commands instead of Beads.
			- Remove tracked `.beads` files only after the `.beans/` state is committed and agent startup behavior is understood.
	- ## [[beadwork]]
		- Source: [jallum/beadwork](https://github.com/jallum/beadwork).
		- [[beadwork]] stores plans, progress, and decisions on a Git orphan branch rather than adding task files to the normal worktree.
		- What it supports:
			- CLI commands for create, show, list, update, close, reopen, delete, comment, label, defer, undefer, and history.
			- Ready and blocked views.
			- Dependencies with `bw dep`.
			- `bw sync` using Git fetch, rebase or replay, and push.
			- JSONL import and export.
			- Agent bootstrap through `bw onboard` and context through `bw prime`.
		- Evaluation:
			- It is compelling if the priority is keeping task state out of the main worktree while still making it Git-native.
			- The orphan-branch model avoids page-tree clutter, which may be pleasant in a Logseq garden.
			- It is conceptually heavier than a plain `.tickets/` directory, and the orphan-branch design means ordinary file browsing is less direct.
		- What migration would take here:
			- Preserve `.beads/issues.jsonl`, then use Beads JSONL as the import source.
			- Run `bw init`, then test `bw import .beads/issues.jsonl --dry-run` if the import accepts Beads-compatible JSONL; otherwise export or transform the open issue manually.
			- Verify `bw list`, `bw ready`, `bw blocked`, and `bw show` for the migrated open work.
			- Run `bw onboard` and convert the resulting guidance into Rulesync-managed project instructions rather than direct generated-file edits.
			- Validate `bw sync` against the repo's remote expectations before removing `.beads`.
			- Remove Beads hooks/config only after confirming the orphan branch contains the desired task history.
	- ## [[ff]]
		- Source: [ff-notes/ff](https://github.com/ff-notes/ff).
		- [[ff]] is a distributed note taker and task manager using RON [[CRDT]]s, with offline-first sync through ordinary file synchronization channels such as Git.
		- What it supports:
			- CLI task and note capture.
			- Conflict-free distributed sync through CRDT/RON semantics.
			- Plain local storage that can be synchronized without a central server.
		- Evaluation:
			- It best fits a personal distributed task/note workflow, not a Beads-like issue tracker for agents.
			- The CRDT approach is technically aligned with parallel edits, but the UX appears bare-bones compared with issue trackers that expose ready, blocked, dependency, and agent-prime concepts.
			- It should be considered only if the goal shifts from "replace Beads" to "experiment with conflict-free personal task notes".
		- What migration would take here:
			- Preserve `.beads/issues.jsonl` as an archive.
			- Manually recreate the open `bd-10` issue as an `ff` task or note, since this is not a direct issue-tracker migration.
			- Decide where `ff` data should live and whether this repo should own that data or merely reference a user-level `ff` store.
			- Avoid deleting `.beads` until the new task is visible from the CLI and recoverable after a Git sync.
			- Update Rulesync-managed task guidance only if `ff` becomes the actual repo workflow.
	- ## [[OpenSpec]]
		- Source: [Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec).
		- [[OpenSpec]] is a spec-driven development workflow for AI coding tools. It creates change folders with proposal, specs, design, and tasks, and then supports apply/archive style workflows.
		- What it supports:
			- Structured proposals, requirements or specs, design notes, implementation tasks, and archives.
			- Agent delivery through commands, skills, or both depending on profile configuration.
			- Core workflows such as propose, apply, sync, and archive, plus expanded workflows such as new, continue, fast-forward, verify, and onboard.
			- A better artifact trail for substantial changes than ordinary chat history.
		- Evaluation:
			- OpenSpec should not be treated as a drop-in task queue replacement. It is stronger for scoped implementation changes than for "what is the next small repo task?".
			- For this garden, it could complement a lightweight tracker: use OpenSpec when a task needs a proposal/design/spec trail, and use a Git-friendly task tracker for ready/blocked repo work.
			- If the desire is to stop maintaining a task database entirely, OpenSpec can absorb larger planned changes, but it does not replace Beads features such as ready lists, issue dependencies, or quick CLI issue updates.
		- What migration would take here:
			- Preserve `.beads/issues.jsonl`, then manually convert the open `bd-10` task into an OpenSpec change only if it deserves a spec/change artifact.
			- Run `openspec init`, choose a delivery mode and workflow profile, and run `openspec update` so generated agent instructions match the chosen profile.
			- Update Rulesync-managed instructions so OpenSpec is described as the spec workflow, not as the universal task tracker.
			- Decide whether a smaller tracker still handles quick tasks, chores, and ready/blocked work.
			- Remove `.beads` only after deciding where small task queue behavior lives.
	- ## Excluded or secondary options
		- [[td]] and [[td/sidecar]] remain useful references, but this garden's definition of [[Task/Management/Git-Friendly]] rejects SQLite-backed task state as not truly Git-friendly for this purpose.
		- [[HamsterBase Tasks]] and [[Will Be Done]] are interesting local-first task apps, but they are GUI or app-sync oriented rather than CLI repo-local Beads replacements.
	- ## Practical migration checklist
		- 1. Preserve `.beads/issues.jsonl` and commit or stash no unrelated work as part of the migration.
		- 2. Pick one replacement for a short trial and import or manually recreate the open `bd-10` task.
		- 3. Verify the new tool's list, show, ready or equivalent, blocked or equivalent, and agent-context commands.
		- 4. Update `.rulesync/` sources for task-management and landing-the-plane behavior; then run `rulesync generate` instead of manually editing generated AI config.
		- 5. Remove tracked `.beads` files, hooks, and Beads-specific README/config only after the replacement state is validated.
		- 6. Record the final change in today's `[[Gard/Diff]]`.
	- ## Bottom line
		- [[tk]] is the most direct Beads-shaped replacement.
		- [[BeansTasks]] is the richest agent-facing flat-file workflow but carries more churn risk.
		- [[beadwork]] is the cleanest if task state should live in Git without living in the working tree.
		- [[ff]] is more of a CRDT task-note experiment than a repo issue tracker.
		- [[OpenSpec]] is worth considering for spec-driven changes, but it should probably complement rather than replace a lightweight task queue unless the repo intentionally gives up ready/blocked issue tracking.
